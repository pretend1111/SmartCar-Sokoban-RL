# 智能车推箱子赛题 — 专用神经网络架构 RFC

> **致 AI 研究者**：本文档纯描述问题（游戏规则、硬件约束、可用监督信号、目标）。**不**对采用何种神经网络架构、状态/动作表示、训练范式作任何主观推荐——这些设计自由度全部留给读者。我们希望基于您对深度学习理论的判断，独立提出适配本场景的架构方案与底层数学论证。
>
> **背景**：本项目是第 21 届全国大学生智能车竞赛"智能视觉组"（推箱子赛题）的算法部分。要在配置受限的微控制器上完成实时决策。

---

## 1. 任务概述

### 1.1 一句话定义

智能车在 16×12 方格地图上完成"推箱子"——把所有"可移动箱子"推到其对应的"目的地"，箱子与目的地按编号 ID 配对。地图中可能含有可炸毁的内墙和炸弹。**用最短时间通关**。

### 1.2 期望产出

一个能在小型 MCU（NXP i.MX RT 系列，无 GPU/NPU）上**实时**给出决策的系统。"决策"在抽象层上是：从当前游戏状态推断"下一步该做什么"。该系统的**输入是游戏当前状态**，**输出能驱动智能车物理动作**的一系列指令。

具体怎么把"状态 → 物理动作"切成几段、各段用什么模型/算法 — **由您设计**。

## 2. 游戏世界完整规则

### 2.1 地图

* 网格大小：**16 列 × 12 行**（X 轴 16, Y 轴 12），共 192 格
* 每格物理边长 20 cm（场地 3.2 m × 2.4 m）
* 单元有 **2 种类型**：
  * **墙** (`#`)：阻挡车、箱子、炸弹
  * **空地** (`-`)：可通行
* 坐标系：以格中心为坐标，左上格为 `(0, 0)`，右下格为 `(15, 11)`
* **外圈墙永久不可摧毁**；内部墙可被炸弹爆炸摧毁
* 每张地图的墙体布局**整局比赛固定不变**（除被炸毁外）

地图 ASCII 示例（实际加载格式）：
```
################          # = 墙
#--------------#          - = 空地
#--$-----------#          $ = 箱子初始位置
#-----.--------#          . = 目的地
#---------*----#          * = 炸弹
#--------------#
#--------------#
#--------------#
#--------------#
#--------------#
#--------------#
################
```

### 2.2 实体

地图上有 4 类动态实体：

#### A. 智能车（player）—— 唯一一个

* 占据网格单元中心，在每个网格中心吸附
* 朝向是 8 个方位之一（45° 倍数）：上 / 右上 / 右 / 右下 / 下 / 左下 / 左 / 左上
* 在物理场地上由真实小车通过 AprilTag 全局定位映射到虚拟世界

#### B. 可移动箱子（box）—— 1~5 个

* 每个箱子有一个 **`class_id`**（0~9 整数）
* 箱子上贴一个图案（10 类卡通角色之一，对应类别 ID）
* `class_id` 在比赛开始时**对参赛者不可见**——必须靠摄像头识别才能知道
* 箱子可被车推动（见 2.3）

#### C. 目的地（target）—— 与箱子数量相等

* 每个目的地有一个 **`num_id`**（0~9 整数）
* 目的地上贴一个数字（0-9）
* `num_id` 同样初始**不可见**，需识别
* 目的地本身**不阻挡通行**——车与箱子可以从上面穿过

#### D. 炸弹（bomb）—— 0~3 个

* 与箱子大小相同，可被车推动
* 推动方式与箱子完全一样
* 当被推到墙上时，会以**接触墙格为中心 3×3 区域爆炸**，摧毁内部墙体（外圈墙不可炸）；炸弹本身在爆炸后消失

#### 配对规则（核心胜利条件）

* 箱子和目的地通过**整数 ID 匹配**：当一个 `class_id == num_id` 的箱子被推到对应的目的地格上时，**箱子和目的地同时消失**（视为配对成功）
* 若 `class_id != num_id`，则箱子可以从目的地格上"穿过"（不消除）
* 比赛胜利条件：**所有箱子消除**

### 2.3 物理规则（移动与推箱）

#### 车的移动

* 车每次移动 1 格（沿 8 方向之一），或原地旋转 45°
* 车不能穿过墙
* 车走到箱子/炸弹格时，**会推动该实体**

#### 推箱链

当车朝某方向走，碰到箱子/炸弹：

* 沿推动方向看下一格：
  * **空地**：箱子/炸弹移到下一格，车跟进
  * **墙**：箱子被阻挡，车也无法移动；炸弹则进入爆炸流程
  * **另一个箱子/炸弹**：递归判断 → 如果链头能动，整链一起动；任一节点被墙阻挡，整链失败
* 链条最大深度无硬限（受地图大小限制）

#### 对角推送（特例）

* **正常推送只支持 4 个正交方向**（上、下、左、右）
* **唯一例外**：当炸弹与墙呈对角相邻、且对角两侧的正交格都为空地时，允许沿对角方向把炸弹推到该墙格 → 触发爆炸（这一规则用于设计需要爆破的死角）

#### 配对消除

每个时间步，引擎检查每个箱子的中心位置是否落在某目的地格内：

* 若 `box.class_id == target.num_id`：箱子和目的地同时移除（配对成功）
* 否则：什么都不发生（箱子继续停留）

### 2.4 部分可观测性 / 视野（FOV）

**核心机制**：车有一个固定 90° 前向视锥（FOV），方向由车的当前朝向决定。

* 箱子和目的地的 ID 信息**只有在该实体进入 FOV 且无遮挡时才能被识别**
* 一旦识别，该实体的 ID 永久记入"已知"集合
* 未识别的实体，参赛者只知道它"位置"但不知道"ID"
* **遮挡**：墙和其他箱子/炸弹会挡住视线
* 实体一旦被识别，再被遮挡也保持已知状态

**ID 排除推理**：若有 N 个同类实体（例如 N 个箱子），ID 取值在 N 个固定整数中：

* 当 N - 1 个实体的 ID 已识别，最后 1 个的 ID 可由排除法推出（不需要再扫描）
* 这一逻辑可由参赛者算法显式实现，也可让模型自己学

### 2.5 终止条件

游戏在以下任一条件成立时结束：

* ✅ **胜利**：所有箱子-目的地全部配对消除
* ❌ **死锁**：场上剩余箱子被推入死角，无法继续推到任何目的地（比赛主动判失败）
* ❌ **超时**：单局 15-25 分钟（具体由组委会设定）
* ❌ **物理违规**：实车穿越外围围墙或意外触发计时器停止

### 2.6 比赛评分

最终成绩 = **实际完成时间** + **罚时**：

* 推错箱子方向：+30 s
* 识别错误：+30 s  
* 漏检箱子：+30 s
* 入库失败（车未停回起点车库）：+15 s

**用时短 + 罚时少 = 胜**。

### 2.7 课程化难度（数据集分级）

我们把训练数据按难度切成 6 个 phase（仅作为数据组织方式，**比赛实际只测最终混合关卡**）：

| Phase | 箱子数 | 是否有炸弹 | 是否有内墙 | 大致解法长度（推数）|
|---|---|---|---|---|
| 1 | 1 | 无 | 几乎没有（空旷） | 3-12 |
| 2 | 1 | 无 | 有内墙 | 4-25 |
| 3 | 2 | 无 | 有内墙 | 6-30 |
| 4 | 3 | 无 | 有内墙 | 10-40 |
| 5 | 1-3 | 有 | 有内墙 | 12-40 |
| 6 | 混合 1-3 + 0-3 炸弹 | 混合 | 15-40 |

每个 phase 当前有 ~1000 张地图（共 ~6000 张），布局多样。

## 3. 智能车的物理能力（决定了控制频率与时延上限）

### 3.1 移动方式

车的物理移动通过电机控制，可执行：

* 前进 / 后退（沿当前朝向）
* 左移 / 右移（垂直于朝向）
* 原地旋转（任意角度）
* 上面的组合（连续模式）

通过虚实结合系统，所有真实运动会被映射到虚拟网格中。当车的网格中心进入相邻格中心时，引擎判定为"移动一格"。

### 3.2 摄像头

车上有一个前向摄像头，用于：

* 接收 360° 大屏的虚拟场景投屏（含全局俯视图 + 第一人称视角）
* 处理目标检测（识别箱子、目的地的图案/数字）
* 提供视觉信息

摄像头帧率与分辨率取决于具体硬件平台（OpenART mini/Plus），见第 4 节。

### 3.3 控制循环

车通过电机闭环（带编码器）实现稳态运动。控制循环典型频率：1 kHz 级（PID/位置环）；上层视觉 + 决策循环远低，受推理速度限制（见第 4 节）。

## 4. 部署硬件平台

参赛所用的微控制器有几种型号，下面是两款常见型号的关键参数：

### 4.1 OpenART mini（基础）

* MCU：NXP MIMXRT1064（Cortex-M7 单核）
* 主频：600 MHz
* 内存：1 MB SRAM + 32 MB SDRAM
* FLASH：4 MB
* **无 GPU、无 NPU**，神经网络推理纯靠 M7 + SIMD（CMSIS-NN 内核）
* 软件栈：MicroPython + OpenMV + TFLite Micro（仅支持 INT8 量化模型）

### 4.2 OpenART Plus（升级款）

* MCU：NXP MIMXRT1176（Cortex-M7 @1 GHz + Cortex-M4 @400 MHz 双核）
* 内存：2 MB SRAM + 64 MB SDRAM
* FLASH：16 MB
* **仍无 GPU/NPU**，但 M4 副核可承担实时控制（电机/PID/IO 闭环）
* 软件栈同上

### 4.3 同一系统上**并发运行**的任务

车上**同一颗 MCU 同时运行**：

1. **传统视觉算法**（OpenMV API）：图像帧捕获、轮廓检测、AprilTag 解码、颜色分割等
2. **YOLO 类目标检测模型**（INT8 quantized TFLite，~100-500 KB 权重）：用于识别箱子图案 / 数字
3. **本文档所讨论的"决策模型"**：本 RFC 想优化的对象

由于 MicroPython + TFLite Micro 单进程串行，三件事在主循环里串行排队执行。**主循环周期目标 ≥ 5 Hz**（即每帧 ≤ 200 ms）。

### 4.4 推理时延 / 内存预算

对决策模型的硬约束（在已扣除 CV+YOLO 后剩余的预算）：

| 项 | 上限 |
|---|---|
| 决策模型权重（FLASH） | 量化后 **≤ 500 KB** |
| 推理时 RAM（tensor arena） | **≤ 2 MB** |
| 单次推理时延（OpenART mini） | **≤ 50 ms** |
| 模型必须能用 TFLite Micro 部署 | 仅可用 op：Conv2D / DepthwiseConv2D / FullyConnected / ReLU / Reshape / Softmax / Concat / Add / MaxPool2D / AvgPool2D。**禁止**：LayerNorm / GroupNorm / BatchNorm（动态）/ LSTM / GRU / Attention（除非可降级为 GEMM） |

> 备注：TFLite Micro 对 LSTM / Transformer 类支持有限。若设计中有循环或 attention，需自证可被低层 op 实现或被替换。

### 4.5 历史经验（避免重蹈覆辙）

K230 平台（6 TOPS NPU + 1 GB RAM，远比 OpenART mini 阔绰）在去年的实战中：**同时加载 2 个 tflite 模型不稳定**，存在 driver / arena 冲突。OpenART mini 各项资源都比 K230 弱，故应优先**单模型 + 必要时手写 INT8 推理**的设计倾向。

## 5. 可用监督信号（用于训练）

我们已经实现以下"专家求解器"作为潜在监督源。它们各自的特征：

### 5.1 求解器 A：IDA*（"exact"）

* 在"推箱-动作空间"上做迭代加深 A\* 搜索（admissible 启发）
* **几乎全局最优**（在时限内若找到解则保证最优性 ± 时限内的不动点）
* 慢：phase 6 难图单次求解可达 60-300 秒
* 给出：完整推送序列 + 总代价（推数 + 车走步数）

### 5.2 求解器 B：加权 A\* / BestFirst（"fast"）

* 同样在推箱-动作空间，但用 weight=1.5 的加权 A\* 启发
* **次优**（在 1.5 × OPT 上界内）
* 快：phase 6 单次求解 5-30 秒
* 给出：完整推送序列

### 5.3 求解器 C：AutoPlayer（启发式）

* 显式实现"探索 → 配对 → 推箱"流水线
* **不保证最优**（典型推数比 IDA\* 多 50-100%）
* 极快：< 1 秒
* 给出：完整低层动作序列

### 5.4 数据集规模与质量

| | 量级 |
|---|---|
| 各 phase 地图数 | ~1000 张 |
| 各地图可探索的"初始 ID 配对数" | 因为 box/target ID 是初始随机分配，每张地图 × 多个 random seed 给出不同的"独立题目"。每张图实际可生成 **N!** 种 ID 配对（N=箱子数）但 box/target 各自洗牌 → 不全独立 |
| 每张图 × 1 种配对 → 1 个 episode → 平均 ~15-30 个 (state, action) 样本 |
| 实际可生成训练样本量级 | 全 phase 总和 **10⁵ ~ 10⁶** 数量级（取决于每张图采样多少种配对） |

### 5.5 局部计算可获取的"领域特征"

下列特征**在任意状态下都能在 ≤ 1 ms 内由经典算法计算**（部署时也能算）。它们**可作为模型输入**：

* **车到任意空地格的 BFS 步数**（O(192) BFS）
* **箱子推送距离场**（每个箱子到对应目的地的最短推送步数，反向 BFS）
* **可达连通分量**（车当前可达哪些空地）
* **死角检测**（哪些格子是箱子推进去就出不来的角）
* **当前已识别 / 未识别实体的视野状态**

是否使用这些 → 由您决定。

## 6. 评估方式（成功指标）

### 6.1 单局成功定义

给定一张地图 + 初始 ID 配对（seed），跑 deterministic 推理。一局 **win** 当且仅当：

* 在 max_steps 步内（每个 phase 不同：30~120）所有箱子-目的地配对完成
* 期间未触发死锁

### 6.2 总体性能指标

* 在每个 phase 的全部 1010 张地图 × 3 个 seed 上跑 deterministic 推理
* 计算 win_rate = wins / episodes

**目标**：

| Phase | 目标 win_rate |
|---|---|
| 1, 2, 3, 4, 5 | **≥ 95%** |
| 6 | **≥ 90%** |

### 6.3 部署约束

模型必须满足第 4.4 节的硬件 / 时延 / 内存约束。

### 6.4 量化损失上限

将 fp32 模型量化到 INT8 后，win_rate 下降 **≤ 2pp**。

## 7. 当前已采用的工程架构（仅作背景说明，**不构成限制**）

为帮助 readers 理解我们当前的工程组织（以判断哪些组件可以复用），简述当前 prototype：

### 7.1 系统拆分

```
[State观测] ──→ [神经网络模型 (BC)] ──→ [一格推箱意图]
                                                ↓
[BFS 路径规划器] ←──────────────────── 把意图翻译成低层动作
       ↓
[引擎 / 实车] ←─── 低层动作（前进 1 格、转 45° 等）
```

* **神经网络只负责"决定下一步推哪个箱子 / 炸弹、推哪个方向"或"先去探索哪个未知 ID"**
* **底层路径规划（车从当前位置走到推箱起点）由 BFS 完成**——不靠模型
* 这种"高层决策网 + 低层经典算法"的拆分**是我们的工程选择**，不是规则限制

### 7.2 当前 baseline 性能（仅供下界参考，**非推荐架构**）

我们目前的 prototype 是一个简单的"hybrid CNN + MLP"——墙体走小型 CNN 编码，实体属性（位置、ID、进度）走 FC，拼接后接 2 层 MLP head 输出离散动作分布；用行为克隆（BC）从 IDA\* / BestFirst 求解器学习；推理时用一个浅层 branch search 缓解 BC 的累积误差。

**当前在 deterministic 推理 + 浅层 branch search 下测得**：

| Phase | win_rate |
|---|---|
| 1 | ~100% |
| 2 | ~99% |
| 3 | ~95% |
| 4 | 40-50% |
| 5 | 60-75% |
| 6 | 50-80% |

我们**怀疑**当前 baseline 的归纳偏置/状态表示/训练范式都次优。**请勿被这些数字反向限制**——它们只用来说明"什么程度是已经做到的"，远未到本场景的能力边界。我们想知道的是：**站在 2024-2025 年 SOTA 的角度，针对这个具体游戏，您会怎么设计**。

### 7.3 您**不必拘泥**的几个工程决策

* **状态如何表示给神经网络** —— 现在是扁平向量 + 部分 2D，您可任意改
* **动作如何输出** —— 现在 54 类 softmax，可改为分层输出 / 回归 / 多目标头
* **是否保留 BFS 底层** —— 可全端到端，可保留经典算法，可任意混
* **是否仅用 BC** —— 可换 DAgger / RL / Q-learning / AlphaZero 自我对弈 / 任何范式
* **是否用单一模型** —— 可用多模型 ensemble、可用世界模型 + 规划

唯一硬约束：**最终能在第 4 节的硬件上、第 6 节的时延内部署，且 win_rate 达到第 6.2 节的目标**。
