## 2. 两个求解器详解

### 2.1 AutoPlayer（模拟真实游玩）

**文件**: `solver/auto_player.py`

模拟比赛中真实的游玩流程，分三个阶段：

```
Phase 1 — 探索
    车在地图中巡游，通过 90° FOV（视野锥）扫描发现箱子和目标的编号
    使用排除法优化：N 个箱子只需扫 N-1 个，最后一个通过排除法推出
    调用 explorer.py 中的 plan_exploration()

Phase 2 — 分析
    匹配箱子→目标配对（class_id == num_id）
    识别被墙阻挡的配对，规划 TNT 开路方案
    调用 bomb_planner.py 中的 analyze_bomb_tasks()

Phase 3 — 执行
    按成本排序配对，逐个推箱子到目标
    使用 push_solver.py 的 bfs_push() 做单箱 BFS 推箱求解
    遇到需要炸弹的配对，先执行 TNT 引爆再推箱
```

**优点**：
- 快速（62 张地图 1.3s）
- 模拟真实比赛流程
- 完整处理探索、推箱、炸弹

**缺点**：
- 逐个推箱，没有全局推序规划
- 可能出现"推完 A 箱挡住了 B 箱的路"的问题
- Phase 4/6 多箱场景步数偏高

### 2.2 MultiBoxSolver（全局最优求解）

**文件**: `solver/multi_box_solver.py`

直接读取完整地图信息（上帝视角），用 IDA*（迭代加深 A*）搜索全局最优推操作序列。

```
状态空间: (车位置, 所有箱子位置, 所有炸弹位置, 已炸墙壁)
搜索目标: 每个箱子到达对应目标位置
启发函数: 每个箱子到目标的最小推送距离之和（反向 BFS 预计算）
```

**关键特性**：
- 不需要探索，直接知道所有编号
- 同时考虑所有箱子的推送顺序
- 有死锁检测（角落死锁等）
- 找到的解是全局最优推序列

**优点**：
- 步数远少于 AutoPlayer（特别是多箱场景）
- 解的质量高

**缺点**：
- 搜索空间指数爆炸，复杂地图可能超时（30s 限制）
- 不模拟真实游玩流程（无探索）
- 速度慢（62 张地图 30.4s）

## 3. 地图体系

### 3.1 地图格式

```
################
#-#---#--#----.#    # = 墙壁
#---#--#-####--#    - = 空地
##-###-#-#-----#    . = 目标位置
#-.#-#-#-#--#--#    $ = 可移动箱子
#-##-#---------#    * = TNT 炸弹
#--#-------##--#
#-*-------*-#--#
#--$-#-----$*--#
#-$--########--#
#----#.--------#
################
```

固定尺寸: 16 列 × 12 行，外圈全是墙。

### 3.2 Phase 分级

| Phase | 地图数 | 箱子 | 炸弹 | 墙壁特征 | 难度 |
|-------|--------|------|------|----------|------|
| Phase 1 | 10 | 1 | 0 | 无内墙（空旷）| ⭐ |
| Phase 2 | 10 | 1 | 0 | 有内墙 | ⭐⭐ |
| Phase 3 | 10 | 2 | 0 | 有内墙 | ⭐⭐⭐ |
| Phase 4 | 11 | 3 | 0 | 复杂墙壁结构 | ⭐⭐⭐⭐ |
| Phase 5 | 10 | 2 | 1 | 有内墙，需炸弹开路 | ⭐⭐⭐⭐ |
| Phase 6 | 11 | 2~3 | 0~1 | 混合所有元素 | ⭐⭐⭐⭐⭐ |

### 3.3 随机种子（Seed）

地图 `.txt` 只定义位置，箱子的 `class_id` 和目标的 `num_id` 在引擎加载时**随机分配**。
- 同一地图不同 seed → 不同配对方案
- `phase456_seed_manifest.json` 记录了已验证可解的 seed

### 3.4 坐标系

- 地图使用 `(col, row)` 表示，col 从左到右，row 从上到下
- 引擎使用浮点位置，格子中心 = `(col + 0.5, row + 0.5)`
- 网格坐标转换: `int(round(x - 0.5)), int(round(y - 0.5))`

### 3.5 离散动作

| 编号 | 含义 |
|------|------|
| 0 | 前进 1 格 |
| 1 | 后退 1 格 |
| 2 | 左移 1 格 |
| 3 | 右移 1 格 |
| 4 | 左转 45° |
| 5 | 右转 45° |
| 6 | 不动 (no-op) |

## 4. 评测工具

### 4.1 benchmark.py

多进程并行评测脚本，一次性跑完所有地图并输出统计报告。

```bash
python benchmark.py                  # 默认 auto 模式，多进程并行
python benchmark.py --solver exact   # 只用 MultiBoxSolver
python benchmark.py --solver fallback # auto 失败回退 exact
python benchmark.py --phase 4        # 只跑 Phase 4
python benchmark.py --save           # 保存结果到 benchmark_results/
python benchmark.py -j 1             # 单进程串行（调试用）
```

### 4.2 preview_failed.py

可视化预览工具，按 Phase 浏览地图求解过程。

```bash
python preview_failed.py --phase 1           # Phase 1
python preview_failed.py --phase 4 --idx 5   # Phase 4 第 6 张
python preview_failed.py --phase 5 --solver exact  # 用精确求解器
```

快捷键: SPACE=播放  N/P=翻页  R=重新求解  ←/→=调速  Tab=渲染  ESC=退出

### 5.3 对比分析

| Phase | AutoPlayer 步数 | MultiBoxSolver 步数 | exact 优势 |
|-------|----------------|---------------------|-----------|
| Phase 1 | 27.5 | 26.5 | -1.0 步 |
| Phase 2 | 32.9 | 31.9 | -1.0 步 |
| Phase 3 | 55.3 | **44.3** | **-11.0 步 (↓20%)** |
| Phase 4 | 109.6 | **41.2** | **-68.4 步 (↓62%)** |
| Phase 5 | 42.8 | 42.5 | -0.3 步 |
| Phase 6 | 85.3 | **47.5** | **-37.8 步 (↓44%)** |

**关键发现**:

1. **步数差距集中在多箱场景**: Phase 4 (3箱) 差距 62%, Phase 6 差距 44%
   - 根本原因: AutoPlayer 逐个推箱没有全局规划
   - Phase 1~2 (1箱) 差距仅 1 步（探索开销）

2. **通关率相同但失败图不同**:
   - AutoPlayer 失败: phase5_09 (炸弹场景), phase6_06 (复杂混合)
   - MultiBoxSolver 失败: phase4_11 (搜索超时), phase6_11 (状态空间太大)

3. **速度差距 23 倍**: AutoPlayer 1.3s vs MultiBoxSolver 30.4s

4. **MultiBoxSolver 不含探索**: 直接读取引擎内部状态，不模拟 FOV 扫描。
   在实际比赛中车必须先探索，所以 exact 的步数是理论下界，无法直接使用。

## 6. 优化方向

基于以上分析，AutoPlayer 的主要优化方向：

### 6.1 多箱推序优化（最高优先级）

**问题**: 逐个推箱导致 Phase 4/6 步数是 exact 的 2~3 倍。
**方案**: 探索完成后，用 MultiBoxSolver 的 IDA* 规划全局推序，再由 AutoPlayer 执行。
**预期收益**: Phase 4 步数从 110 降至 ~50, Phase 6 从 85 降至 ~50。

### 6.2 修复失败地图

**phase5_09.txt**: AutoPlayer 的炸弹规划逻辑有 bug。
**phase6_06.txt**: 复杂墙壁结构下 AutoPlayer 匹配/推送失败。

### 6.3 步数异常地图

- **phase4_06/07**: 289/276 步 → compact 布局下推序极差
- **phase6_11**: 231 步 → 混合场景推序+炸弹规划都有问题
