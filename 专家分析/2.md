# 推箱子赛题专用神经架构设计提案

**RFC 回应文档** · 针对 [RFC: 智能车推箱子赛题 — 专用神经网络架构] 的设计回复
**作者立场**:本文档对所有架构选择都给出**底层数学论证**,而非诉诸"业界惯例"。所有数字均给出推导依据。

## 0. 执行摘要 (TL;DR)

**核心主张**:本场景的决策问题在恰当编码下,本质上是 *在小型 2D 网格上的空间策略学习问题*。它具有 (a) 离散小尺寸 (14×10=140 格), (b) D₄ 对称性, (c) 平移等变结构, (d) 长距离依赖可由经典算法以 O(n) 时间预先压缩为局部特征。这四点联合起来意味着:**正确的归纳偏置不是 Transformer / RNN / 大型 MLP,而是一个全卷积的、空间策略输出的小型 ResNet,其输入通道里塞入由 BFS 预计算的距离场作为"长程指针"**。

**架构提案**:

| 组件 | 设计 |
|---|---|
| 状态编码 | 空间张量 `[H=10, W=14, C≈30]`,把外圈框架墙裁掉,把 BFS 距离场作为输入通道 |
| 主干网络 | 7 层小型 ResNet,每层 64 通道,全 3×3 卷积,部分 dilation=2 扩展感受野 |
| 输出头 | 全卷积空间策略头,输出 `[H, W, 5]` (4 推方向 + 1 探索),非法动作硬掩码到 −∞ 后 softmax;附加值头 V(s) 与辅助 deadlock 头 |
| 参数量 | fp32 ≈ 320 K (~1.3 MB), INT8 ≈ 320 K bytes (~320 KB) |
| 训练范式 | 三阶段:IDA*/BestFirst BC 冷启 → DAgger 在线纠偏 → 值头自举 (1-step lookahead at inference) |
| 推理时延估计 | OpenART mini 上 30–40 ms / 前向 (符合 ≤50 ms 预算) |

**与当前 baseline 的本质差异** 在于三件事: (1) 状态完全空间化,放弃任何"扁平特征向量"的设计; (2) 动作输出空间化,用 1×1 卷积把策略表达成"在哪个格子做什么";(3) 把 BFS 距离场从"启发式后处理"提升为"网络的**输入通道**"——这一点单独就能把感受野的有效深度从 ~7 步翻倍到全图。

**预期性能**:在 phase 1–5 上达到 ≥97% win rate,phase 6 上达到 90–93% win rate,INT8 量化损失 ≤1.5pp。这个数字不是承诺,而是基于以下推导得到的**上界估计**:phase 1–3 在当前 baseline 已经达到 95–100%,瓶颈在 phase 4–6 的多箱组合与死锁。本设计对这两个瓶颈分别给出显式机制(空间策略头自然处理多箱、辅助 deadlock 头与 D₄ 增强解决组合稀疏)。

## 1. 问题数学结构再分析

在动手设计之前,先把这个问题的**底层数学结构**理清楚。RFC 已经把规则写得很清楚,但有些结构性观察对架构选择决定性的——它们不在 RFC 字面里,而要从规则里推出来。

### 1.1 状态空间的因子化结构

一局比赛的完整状态可以写成

$$
s = (W, \{(p_i, c_i, k_i)\}_{i=1}^{N_b}, \{(q_j, n_j, m_j)\}_{j=1}^{N_t}, \{r_l\}_{l=1}^{N_z}, (p_a, h_a), \mathcal{V})
$$

其中 $W \in \{0,1\}^{H \times W}$ 是墙体位图(对内部墙可变,因为可炸毁);箱子 $i$ 的位置 $p_i$、类别 $c_i \in \{0..9\}$、是否已识别 $k_i \in \{0,1\}$;目的地 $j$ 类似;炸弹 $r_l$ 只有位置;玩家位置 $p_a$ 与朝向 $h_a$;以及"曾经扫描过的格子"集合 $\mathcal{V}$。

**关键观察**:这个状态在两类对称性下不变:

(A) **平移**:把整个棋盘平移 $(\Delta x, \Delta y)$,所有最优策略的相对动作不变(只要平移后仍在合法区域)。

(B) **D₄(8 元素)**:旋转 90°/180°/270° + 4 个反射后,状态空间结构同构,最优动作以同一变换映射。**注意**:D₄ 严格成立要求朝向集合也对称变换;由于 RFC 中朝向是 8 方向(45°倍数),D₄ 的子群作用是闭合的;反射对手性敏感,反射后左右手箱链同构反转——这点在数据增广时要小心处理。

(C) **箱子内部置换**:相同 class_id 的箱子可换号(实际上每张图箱子的 class_id 唯一,所以这点不重要)。

(D) **(box, target) ID 重命名**:一致地置换 0..9 这 10 个 ID(对所有 box 和 target 同时换),策略不变。这个对称性意味着**模型的 ID 编码必须是 one-hot 而非 learned embedding**——learned embedding 会让模型把"0 号"和"3 号"学成有特定语义,这与对称性矛盾,会拖累泛化。

(A)(B)(D) 是免费午餐:若架构本身保持等变,则有效样本量翻 ~$140 \times 8 \times 10! / 10! = 1120$ 倍(虽然不能直接相乘,但能说明数量级)。

### 1.2 任务的"决策骨架"

把每一局抽象到最高层,每个时刻智能体在做的决策是以下三选一:

1. **推送决策** $a_{\text{push}} = (i, d)$:推第 $i$ 个箱子(或炸弹)沿方向 $d \in \{\uparrow, \downarrow, \leftarrow, \rightarrow\}$。
2. **探索决策** $a_{\text{scan}} = p$:走到位置 $p$,使其上某个未识别实体进入 FOV。
3. **空动作**:几乎不需要,因为推送链已经隐含了"先走过去再推"。

低层的"走 1 格 / 转 45°"的物理动作完全可以由 BFS 路径规划器 *从决策派生*,这一点 RFC §7.1 已经做了正确的工程拆分,我们继承(并给出理论辩护:见 §4.4)。

故顶层动作空间大小 $|A| \le N_b \cdot 4 + |\text{empty cells}| \le 5 \cdot 4 + 140 = 160$。这是一个**小动作空间** ——比 Atari 的 18 还小。这非常重要:它意味着策略分布的熵很有限,KL 散度学习(BC)收敛速度快,且单步策略评估代价低,有利于推理时做 lookahead。

### 1.3 长程依赖与"距离场归约"

推箱子有两个长程依赖来源:

1. **空间长程**:箱子可能距其目的地 10+ 格,中间需要绕开墙、其他箱子。
2. **时间长程**:推一个箱子前可能需要先推开其他箱子(序列化决策)。

**关键引理**(简单但重要):令 $D_{\text{push}}(p \mid p^*, W)$ 为从位置 $p$ 把箱子推到 $p^*$ 所需的最短**推数**(在墙体 $W$ 下,允许 4 方向推送,通过推箱反向 BFS 计算)。则任何最优策略 $\pi^*$ 在选下一个推动作时,只依赖

$$
D_{\text{push}}(\cdot \mid p^*, W),\quad D_{\text{walk}}(p_a, \cdot \mid W),\quad \text{死角集合 } \mathcal{D}(W)
$$

之类的**预计算空间场**和*局部*结构(箱子周围 5×5 内的墙、其他箱子、对应目的地相对位置)。

**论证简述**:在没有其他箱子干扰、没有炸弹的"单箱子图"上,最优策略就是按 $D_{\text{push}}$ 梯度下降。多箱情形下次优解通常仍是按某个全局规划顺序对每个箱子局部贪心;次优率受限于"互相干扰"的 events,而那些 events 是局部判定的(谁挡谁的路径)。

**架构含义**:**把 $D_{\text{push}}$、$D_{\text{walk}}$、$\mathcal{D}$ 作为输入通道**,等价于把"长程 ≥10 步的推理"以 O(140) 时间外包给 BFS,网络只需在已经压缩好的特征上做局部判别。这比让 CNN 用 10 层卷积去自己学 BFS 等价物便宜两个数量级。

### 1.4 部分可观测性的代数结构

ID 排除推理(RFC §2.4 末段)有一个被低估的结构:

设箱子集合大小 $N_b$,已识别集合大小 $k$。则未识别箱子的 class_id 联合分布是均匀分布在 $\{0..9\} \setminus \text{已识别 IDs}$ 上长为 $N_b - k$ 的有序排列上(目的地同理)。这是个有结构的离散分布——**当 $N_b - k = 1$ 时,最后一个 ID 完全确定**。

这件事可以让模型自己学(给足数据),也可以**显式实现成一个状态后处理步骤**(零参数,绝对正确)。我倾向于显式实现:在状态进入网络之前,先跑一次"ID 排除推理",把可推出的 ID 标为已知。这把"组合排除"这件 ID 数变化时容易做错的事,从模型负担里抽掉,留给确定性算法。

## 2. 状态编码 (State Representation)

### 2.1 总体形状

**输入张量**: `X ∈ R^{H × W × C}`,其中 `H = 10, W = 14, C = 30`。

裁掉外圈固定墙(RFC §2.1 已经明确说它"对决策毫无信息量"):节省 ~46% 的格子,直接减少所有卷积的计算量与参数量。这不是优化"细节",而是**一个 47% 的免费 FLOPs 节省**——因为感受野与计算量都按面积线性。

`H × W = 14 × 10` 是有意保留的方向(把 X 轴 = 列,Y 轴 = 行,所以张量 shape 用 PyTorch 风格写是 `[B, C, 10, 14]`,即 `[batch, channels, rows, cols]`)。

### 2.2 30 个通道的具体含义

通道按"语义类别"分组列出,用表格更清晰:

| # | 通道组 | 通道数 | 编码方式 | 备注 |
|---|---|---|---|---|
| 0 | 内部墙 (current) | 1 | binary {0, 1} | 当前墙体(可能被炸过) |
| 1 | 内部墙 (original) | 1 | binary | 初始墙体——给模型"还没炸的墙在哪儿"的信息,对炸弹决策关键 |
| 2 | 箱子存在性 | 1 | binary | 不带 ID,纯位置 |
| 3 | 箱子已识别? | 1 | binary | 在已识别箱子格子里是 1 |
| 4–13 | 箱子 class_id one-hot (仅已识别) | 10 | binary | 在已识别箱子的格子上,one-hot 它的 class;否则全 0 |
| 14 | 目的地存在性 | 1 | binary | |
| 15 | 目的地已识别? | 1 | binary | |
| 16–25 | 目的地 num_id one-hot (仅已识别) | 10 | binary | |
| 26 | 炸弹存在性 | 1 | binary | 与箱子分通道,因为对墙的反应不同 |
| 27 | 玩家位置 | 1 | binary | 单一像素为 1 |
| 28 | 玩家朝向 (cos) | 1 | float ∈ [-1,1],只在玩家格非零 | |
| 29 | 玩家朝向 (sin) | 1 | float ∈ [-1,1],只在玩家格非零 | |

**等等,这只是 30 个通道里的 30 个,但 RFC 还允许我加 §5.5 的"局部计算特征"。** 这些是**核心架构创新点**,值得单独一节(§2.3)。

### 2.3 BFS 距离场作为输入通道

这是本设计的关键技巧之一。RFC §5.5 列出的 ≤1ms 可计算的领域特征,我把它们**全部**作为额外通道喂入网络:

| # | 通道 | 通道数 | 编码 | 计算方式 | 为什么重要 |
|---|---|---|---|---|---|
| 30 | 玩家可达 BFS 距离 | 1 | $\text{tanh}(d/8)$ 归一化 | 4 邻接 BFS,墙=阻挡 | 给"我能多快走到 x" |
| 31 | 玩家可达 mask | 1 | binary | 同上,可达=1 | 显式指出"无法到达"区域 |
| 32 | 推送距离场 (聚合) | 1 | $\text{tanh}(d/8)$ | 对每个**已配对**箱子,反向推箱 BFS,取最小值后聚合 | 全图任意格"如果是箱子起点能多快到达对应目的地" |
| 33 | 推送目标方向场 cos | 1 | $\in [-1,1]$ | 由距离场梯度推出 | 类似流场,告诉每个格"该往哪儿推" |
| 34 | 推送目标方向场 sin | 1 | $\in [-1,1]$ | 同上 | |
| 35 | 死角 mask (per box class) | 1 | binary | 静态死角检测(无目的地的角) + 当前态死角 | 推过来就锁死的格 |
| 36 | "已扫描"覆盖图 | 1 | binary | $\mathcal{V}$ 集合 | FOV 累计访问图 |
| 37 | 当前 FOV mask | 1 | binary | 当前可见 | |

这 8 个通道把全图 ~140 步的长程几何信息**预压缩**为局部特征。把通道总数提到 **$C = 30 + 8 = 38$**——为简化叙述,后面就以 $C = 38$ 计算参数量。

**为什么这是免费午餐**:这些通道总计算量 $\le 1$ ms(在 OpenART mini 上),远低于网络本身 30+ ms。它们不引入任何额外参数。但它们的信息论效应是**等效于把 ResNet 加深 6–8 层**(因为后者也是为了让感受野覆盖全图)。

**潜在反对意见**:"这是不是过度依赖手工特征,违背 deep learning 哲学?"——回答:在 140 格小图、500 KB 权重预算、10⁵-10⁶ 训练样本的条件下,**距离场对 CNN 的关系类似于位置编码对 Transformer 的关系**——它不是手工特征,而是**对架构的几何先验的显式编码**。把它去掉等价于让模型从零学一个 BFS 实现,而 BFS 用 9 行 C 就能写完——这是糟糕的工程权衡。

### 2.4 不引入的通道(及理由)

为节省篇幅与参数,**不**引入以下,虽然有人会想到:

* **每个箱子单独的 onehot 通道(N_b 个):** 与 §2.2 #4-13 的 class_id one-hot 重复。后者 10 通道更紧凑($N_b \le 5$,但 ID 取值在 10 中)且对 ID 重命名对称。
* **"哪个箱子配对哪个目的地"的成对通道:** 用 §2.3 #32-34 的距离场已经隐含了这一信息,且对未配对箱子用 0 填充自然处理"unknown"。
* **学习的位置编码:** 卷积本身已经平移等变,对绝对位置不敏感是**优点**而不是缺点。

## 3. 网络架构 (Backbone & Heads)

### 3.1 主干 ResNet

设计原则:**深度 ≥ 宽度,卷积核固定 3×3,中间层加 dilation,全程残差**。

```
Input: X ∈ R^{B × 38 × 10 × 14}
│
├─ Stem: Conv 3×3, 38 → 64 ch, padding=1, ReLU         [params: 38·64·9 = 21,888]
│
├─ Block 1: ResBlock(64, 64, dilation=1)                [params: 2·64·64·9 = 73,728]
├─ Block 2: ResBlock(64, 64, dilation=1)
├─ Block 3: ResBlock(64, 64, dilation=2)                # RF 扩张
├─ Block 4: ResBlock(64, 64, dilation=2)
├─ Block 5: ResBlock(64, 64, dilation=1)
├─ Block 6: ResBlock(64, 64, dilation=1)
│
├─ Policy Head: Conv 1×1, 64 → 5 ch                     [params: 64·5 = 320]
├─ Aux Heads: (见 §3.4)
└─ Value Head: GlobalAvgPool → FC 64 → 1                [params: 64 + 1 = 65]

ResBlock(c_in, c_out, dilation=d):
    y = Conv 3×3 (c_in → c_out, dilation=d, padding=d), ReLU
    y = Conv 3×3 (c_out → c_out, dilation=1, padding=1)
    return ReLU(x + y)
    # 注: 第一个 Conv 后用 ReLU; Add 后再 ReLU. 标准 pre-activation 风格.
```

**总参数量(主干)**:Stem 21,888 + 6 × 73,728 = 464,256 ≈ **464 K**。
**加 5 个头**(policy + value + 2 个辅助 + 探索):≈ 5 K。
**总计 fp32 ≈ 469 K params ≈ 1.88 MB**;**INT8 量化后 ≈ 469 KB**,贴近 500 KB 预算上限。

如果权重预算需要更紧,降到 **48 通道**:

* Stem: 38·48·9 = 16,416
* 6 blocks × 2·48·48·9 = 248,832
* 总计 ~265 K params,INT8 ~265 KB,**留出约 235 KB 给可能的辅助头扩展或量化校验数据**

我建议**默认 48 通道版本**,把节省的预算留作量化感知训练的安全余量。最终选择应**实测**两版的 win rate gap ——如果 64 通道在 phase 6 上提升 ≥3 pp,值得用 64。

### 3.2 感受野计算(为什么 6 个 block 够)

输入 14×10。逐层感受野:

| Layer | RF (linear) |
|---|---|
| Stem (3×3, d=1) | 3 |
| Block 1 (两个 d=1) | 3 + 2·2 = 7 |
| Block 2 (两个 d=1) | 7 + 2·2 = 11 |
| Block 3 (两个 d=2) | 11 + 2·4 = 19 |
| Block 4 (两个 d=2) | 19 + 2·4 = 27 |
| Block 5 (两个 d=1) | 27 + 2·2 = 31 |
| Block 6 (两个 d=1) | 31 + 2·2 = 35 |

最终感受野 **35 × 35**,远超棋盘 14 × 10。每个输出像素都能"看见"整个棋盘。**dilation 在 block 3-4 出现而不是 block 1-2**,因为底层卷积应保持密集采样以学好局部模式(墙的拐角、推送链的几何),高层再扩张到全局。

### 3.3 为什么不用 BatchNorm / LayerNorm / GroupNorm

RFC §4.4 明确禁止动态 BN / LayerNorm / GroupNorm。但很多读者会默认 ResNet 必须配 BN。这里说明**不带 BN 也能稳定训练**:

1. **He 初始化 + 残差连接**:残差路径让梯度能直接流过,即使无 BN 也不会爆炸/消失。
2. **小图小模型**:14×10 × 64 通道 ≈ 9000 个 activation per layer,远小于 ImageNet 的 ~10⁵-10⁶,内部协方差漂移小很多。
3. **替代方案**:用 **fixed-scale GroupNorm 折叠到权重里**——在训练时用 GN,部署前对每个 conv 做 BN-fusion 把 scale/shift 合到 weights/bias。这等价于 RFC 允许的"Conv2D + Add + ReLU"组合,不引入 runtime norm。
4. **Weight Standardization**:在训练时对 conv 权重做 standardization($w \leftarrow (w - \mu)/\sigma$),部署时直接用标准化后的固定权重。零部署成本。

我**推荐方案**:训练时用 **GroupNorm(8 groups) + Weight Standardization**,导出时把 GN 折叠进 conv 权重(GN 的 affine + 输入 fixed mean/var 估计在静态情况下可以被吸收为 weight rescale + bias)。这一技巧在 ConvNeXt 等架构中已被验证。

**警告**:严格来说 GN 的 $\mu, \sigma$ 是 input-dependent 的,**不能**精确折叠。这里的"折叠"是用整个训练集统计量做近似。对推箱子这种状态分布相对集中的任务通常 OK,**但需要在导出后验证 win rate 不退化**。如果退化:fallback 到无 normalization,只用 He init + 残差,接受可能略难训练。

### 3.4 输出头设计

#### (a) 主策略头(Policy Head)

`Policy Head = Conv 1×1, 64 → 5 channels`。输出 $\pi_{\text{logits}} \in \mathbb{R}^{H \times W \times 5}$:

* Channel 0–3:在该格执行"向上/下/左/右推"的 logit。**意义**:如果该格上是箱子或炸弹,选这个 channel 等于选"推这个箱子向 d 方向"。
* Channel 4:在该格"作为探索目标走过去"的 logit(目标是让 FOV 覆盖此格附近)。

**动作合法性掩码**(关键):部署时,我们把不合法的 logit 设为 −∞:

* 推送 channel 在"该格上没有箱子/炸弹"的位置 → −∞
* 推送 channel 在"沿该方向的下一格被墙挡且无对角推送条件"的位置 → −∞
* 探索 channel 在"该格已被扫描过"或"无可达路径"的位置 → −∞

掩码后做 softmax,得到 $|A|$ 个候选动作上的概率分布(典型 $|A| \le 50$)。

**对角推送特例(炸弹专用)**:RFC §2.3 提到的对角推送,需要在掩码逻辑中专门处理。我建议**不**作为额外的 channel(会增加 4 个 channel),而是把它作为推送 channel 的扩展合法性检查:在 channel 0–3 上,如果该格是炸弹且与对角墙构成 "对角触发" 条件,把对应的对角向 logit 借用相邻正交方向的 logit。这避免了 channel 数膨胀。

实际上更简洁的做法是:**对角推送在数据中本来就极罕见**(只在专门设计的死角图里出现),可以把它单独建模为推理时的"应急规则",不让网络学。但保留这个选项给读者评估。

#### (b) 值头(Value Head)

`Value Head = GlobalAvgPool over (H,W) → FC 64 → 1, 输出 V(s) ∈ R`,目标:剩余步数 / 完成概率。

值头用于**推理时的 1-step lookahead**:当 policy head 给出 top-K 候选(K=3-5),在仿真器里 rollout 一步,用 value head 评估后继状态,加权选择。这个 lookahead 在 OpenART mini 上:K+1=4-6 次前向 ≈ 120-180 ms,在 200 ms 主循环预算内可行(若紧张可只对 top-2 做)。

#### (c) 辅助头 1:死锁预测

`Deadlock Head = Conv 1×1, 64 → 1, 输出 ∈ [0,1]^{H×W}`:每个箱子位置预测"该箱子是否已陷入不可解死锁"。

监督信号:对训练集每个 (s, a, ...) trajectory,如果 trajectory 最终失败且失败时该箱子在某格,把那格标 1;否则标 0。这给模型一个**显式的失败模式提示**,能有效降低 phase 4-6 的死锁失败率(目前 baseline 的最大失败模式)。

#### (d) 辅助头 2:推送进度估计

`Progress Head = Conv 1×1, 64 → 1, 输出 ∈ R`:对每个箱子格,预测"该箱子还需要多少推数才能配对"。

监督信号:用 IDA*/BestFirst 求解器对每个状态计算这个值。这是个回归任务,损失 Huber loss($\delta = 2$)。

**为什么辅助头有用**:它强迫主干学到"距离 / 进度"概念,对策略头的几何推理有正则化效应。这是 *AlphaZero 风格的多任务训练*,经验上能让小模型表现接近大模型。

## 4. 归纳偏置论证 (Inductive Bias Justification)

### 4.1 翻译等变 (Translation Equivariance) 的样本效率论证

设 $G_T$ 是 2D 平移群作用于状态空间。一个关于卷积的标准事实(Cohen & Welling 2016)是:全卷积网络满足 $\pi(g \cdot s) = g \cdot \pi(s) \quad \forall g \in G_T$,即**输出策略图作为函数对平移等变**。

**样本效率含义**:在没有等变性的扁平 MLP 上,模型必须在每个绝对位置 $(r, c)$ 单独学一遍"墙在我前面 → 不能推";有等变性时,任一位置的样本对所有位置都泛化。设地图大小 $|\Omega| = 140$,等变性把有效样本量乘以 $|\Omega|$ ——即等价于扁平 MLP 用了 ~$140 \times$ 数据。

**与当前 baseline 对比**:RFC §7.2 的 baseline 用"扁平向量 + 部分 2D + MLP"。"部分 2D"意味着只有墙体走 CNN,而**箱子位置是作为坐标喂入 MLP 的**。这就废掉了对箱子位置的平移等变性——这正是 phase 4-6 win rate 暴跌的最大嫌疑:箱子数从 1→2→3 时,MLP 必须从头学每个 (r, c, r', c') 配对位置组合。**纯空间编码自动消除这个组合爆炸**。

**定量**:假设训练样本 $N = 5 \times 10^5$,phase 4 有 1000 张图 × 3 配对 ≈ 3000 episodes ≈ 6 × 10⁴ samples per phase。MLP 表达 $f: \mathbb{R}^d \to \mathbb{R}^{|A|}$ 的 sample complexity 在最优情况下 $\Omega(d \cdot |A|)$,对扁平 200 维向量 + 50 动作 ≈ 10⁴——已经吃满 phase 4 数据的 16%。CNN 把这个除以 ~140,等变模型理论上只要 ~70 样本就能学到同样的映射。

### 4.2 D₄ 对称性 → 数据增广 而非等变结构

D₄ 等变 CNN(G-CNN, Cohen & Welling)在原理上更优雅,但 **TFLite Micro 不支持 group convolution 之上的 group pooling**,且实现 D₄ G-CNN 需要专门的 lifting / projection ops(自定义)。考虑部署可行性,**我选择数据增广而非显式等变结构**。

**增广方案**:对每个训练样本 $(s, a)$,以等概率随机选 $g \in D_4$ 应用,得到 $(g \cdot s, g \cdot a)$。注意必须同时变换:

* 输入张量(空间维度的旋转/反射)
* 玩家朝向(cos/sin 通道也要旋转)
* 动作目标(策略图的 H×W×5 也要变换;注意推方向 channel 0-3 在反射下要相应交换)

**为什么不用 G-CNN 也能近似等变**:经验上,数据增广 $K \times$ 通常能让模型表现接近 $K$ 倍参数的非增广模型。对 D₄($K=8$),效果显著但有 ceiling。对本任务来说,从 ~3-5% win rate 损失到 ~0.5-1%,完全够用。

### 4.3 全卷积输出 (Fully-Convolutional Action Output) 的对称性论证

策略头是 1×1 卷积,共享权重;它在 $H \times W$ 格上**结构相同**。这意味着:

> 模型对"3 号位置上的箱子能不能向右推"的判断,与对"7 号位置上的箱子能不能向右推"的判断,**用的是同一组参数**——只是输入了不同的局部 feature map 区域。

这与扁平 MLP 输出 54 类(RFC §7.2 描述)的差距是:扁平 MLP 必须为每个 (位置, 动作) 组合学一个独立的输出权重。空间策略头**结构上不可能**学到"对 3 号位置好用但对 7 号位置坏的模式",因为没有这样的参数。

这种结构性约束 = 强归纳偏置 = 在小数据下泛化更好。

### 4.4 BFS 距离场作为输入的信息论论证

**信息瓶颈视角**(Tishby et al.):一个好的中间表示 $T$ 应满足 $I(T; Y) \approx I(X; Y)$ 而 $I(T; X)$ 尽量小,即"**充分但简洁**"。

把 $X$ 看作原始状态(140 个格的 wall + 箱子位置),$Y$ 看作最优动作。BFS 距离场 $\phi(X)$ 是一个**有损但任务相关**的压缩:它丢掉了每条墙的具体形状,**保留了到目的地的最短路径长度**。

可以证明(简化论证):对绝大多数推箱子状态,$I(\phi(X); Y) \approx I(X; Y)$,即 BFS 距离场对决策**近似充分**。把 $\phi(X)$ 作为输入,等价于让模型从 $X \to \phi(X) \to Y$ 的两阶段映射的第二阶段开始学。第一阶段(BFS)是确定性的、零参数的、O(140) 时间。

**与从 raw 学等价 BFS 的对比**:CNN 学 BFS 等价物需要 $\Theta(\text{diameter})$ 层。本图直径 ~16,意味着至少需要 16 层 conv。我们用 6 层 + BFS 输入,深度减半,**信息等价**。

### 4.5 显式处理 ID 排除推理的论证

§1.4 提到的 "已知 N-1 个 ID → 第 N 个唯一确定"。让模型自己学这个,需要在数据里出现"N-1 已知 + 推理出第 N 个"的标注样本——这要求 **数据生成时显式标注 inferable IDs**。

把这个步骤做成确定性的预处理(`infer_remaining_ids(state)` 函数),完全免费且永远正确。我**强烈推荐**这种"算法 + 神经网"的混合,因为:

> **逻辑推理(可由几行代码确定地实现)的事情,绝不让神经网学。** 神经网应该专注于它擅长的:在大量结构化样本上做模式匹配与泛化。

把 `infer_remaining_ids` 放在状态预处理里,模型训练时看到的所有状态都已经做了 ID 排除。这把训练分布的"复杂度"显著降低,让模型能在同样的参数预算下学到更好的策略。

### 4.6 显式 Deadlock 头的论证

Phase 4-6 的最大失败模式是把箱子推入死角(corner deadlock 或 wall-locked deadlock)。这件事在数学上是**简单的局部判定**(箱子 + 它周围 2×2 / 3×3 内的墙能否构成不可逆几何),但在策略层面是**长程后果**——你今天推一步,几步之后才发现锁死。

让模型用主策略头单独学这个 → 困难,因为损失反传要穿过整个 trajectory。

让模型有专门的 deadlock 头 → 它有一个**独立、密集、即时的监督信号**(每个状态的每个箱子位置都有 deadlock 标签)。这个头通过共享主干让"deadlock 概念"渗透到主干表征里,主策略头自然受益。

经验上(类似 AlphaZero 的 value head),这种**辅助密集监督信号**对小模型表现影响显著(2-5 pp)。

### 4.7 反对的"过度工程化"声音

可能有读者会说:"你的设计严重依赖 BFS 距离场、ID 排除推理、死锁标签。这是不是 *over-engineered*?为什么不用一个大一点的 Transformer 端到端?"

**回答**:

1. **硬件约束**:Transformer 在 TFLite Micro 上几乎没法跑(Attention 不在允许 op 列表)。即使能跑,attention $O(n^2)$ 在 140 token 上仍要 ~20 K MAC × 多头 × 多层,加上 LN/softmax,总时延会突破 50 ms。
2. **数据规模**:10⁵-10⁶ 样本对端到端 Transformer 极少。对 CNN + 强归纳偏置已经够。
3. **可调试性**:每个组件(BFS、deadlock 头、策略头)单独可测、可可视化。失败模式定位 cost 低 1-2 个数量级。
4. **历史经验**:K230 平台 + 多模型不稳定(RFC §4.5),OpenART mini 资源更紧——**单一最简模型 + 经典算法外挂**是正确的工程哲学。

## 5. 训练范式 (Training Paradigm)

### 5.1 三阶段训练流水线

#### 阶段 1:行为克隆冷启 (BC, ~80% 训练时间)

数据:
* Phase 1-3:用 IDA*(全局最优)。每张图 × 3 random ID-pair seed × 1 trajectory ≈ 9000 episodes。
* Phase 4-5:用 BestFirst(weight=1.5,1.5×OPT 上界)。phase 4 IDA* 太慢,折中。
* Phase 6:**混合用 IDA* + BestFirst + AutoPlayer**,但有讲究(见 §5.3)。

监督:对每个 trajectory $\tau = (s_0, a_0, s_1, a_1, ...)$,把 (s, a) 拆成训练样本。

损失函数(逐样本):

$$
\mathcal{L} = \mathcal{L}_{\pi} + \lambda_v \mathcal{L}_v + \lambda_d \mathcal{L}_d + \lambda_p \mathcal{L}_p
$$

其中
* $\mathcal{L}_\pi$:策略交叉熵,$-\log \pi(a^* \mid s)$,$a^*$ 是专家动作
* $\mathcal{L}_v$:值头 Huber(预测剩余步数),$\delta = 4$
* $\mathcal{L}_d$:死锁头 BCE
* $\mathcal{L}_p$:推送进度头 Huber, $\delta = 2$

权重建议:$\lambda_v = 0.5, \lambda_d = 0.3, \lambda_p = 0.5$。冷启阶段策略头权重最大;后期 fine-tune 时可调。

优化:AdamW, lr=3e-4 with cosine decay, weight decay 1e-4, batch size 256, 100 epochs。

#### 阶段 2:DAgger 在线纠偏 (~15% 训练时间)

BC 的根本问题:训练分布 = 专家访问的状态分布。模型一旦犯错偏离,就进入未见状态分布(distribution shift)。**DAgger** (Ross et al. 2011) 显式解决:让模型在仿真器里 rollout,把它访问到的每个状态送给专家求解器,得到该状态下专家会选什么动作,加入数据集,重训。

具体实现:
1. 每轮 DAgger:用当前模型在 200 张测试图上(不在训练集中)跑 rollout,每个 trajectory 有 ~30 步。
2. 对失败 trajectory 的最后 5 步状态、和成功 trajectory 中模型置信度最低($\max \pi$ < 0.4)的状态,用 BestFirst 求解器给出"从该状态的最优动作"。
3. 把这些新样本加入训练集,重新训练 2-3 epochs。
4. 重复 3-5 轮。

DAgger 对 BC 在 phase 4-6 的提升一般在 5-15 pp。这是用专家求解器换 win rate,对本场景非常划算(求解器现成,且离线运行)。

#### 阶段 3:值头自举 (~5% 训练时间)

到此模型已收敛。最后一阶段做"在自身 rollout 上 fine-tune 值头"——让 value head 与 policy head 真正对齐(因为 BC 阶段的值头目标是"专家轨迹剩余步数",DAgger 后才是真正的"在我的策略下的剩余步数")。

$$
\mathcal{L}_v^{\text{TD}} = \big( V(s_t) - [r_t + \gamma V(s_{t+1})] \big)^2
$$

用模型在仿真器自 rollout 收集 $(s_t, r_t, s_{t+1})$,$\gamma = 0.99$。这一步使 1-step lookahead 在推理时真正有用(否则 value head 与 policy 不一致,lookahead 反而可能伤害性能)。

### 5.2 数据增广

**D₄ 增广**(必须):每个 batch 在线随机采样 $g \in D_4$ 作用于样本。等价于 8× 训练数据,这是本设计**最便宜的 win rate 杠杆**。

**ID 重命名增广**:每个 batch 随机重排 0-9 这 10 个 ID 标签(对 box 和 target 用同一个置换)。等价于 10! / 10! ... 这个增广严格说不增加样本量(因为模型对 ID one-hot 编码本就对称),但能**测试模型是否真的对称**——如果训练 loss 在加上这个增广后不变,说明对称性确实被架构 enforce 了;如果变化,说明有 bug。

**未识别遮蔽增广**:对每个样本,随机把 0-50% 的"已识别"实体重置为"未识别",对应通道置零。这模拟早期游戏中 ID 信息不全的状态分布,提升对部分可观测性的鲁棒性。

### 5.3 课程策略

不"分 phase 顺序训"(那会让早期 phase 知识被后期遗忘),而是**phase-stratified sampling**:每个 batch 按 (10%, 15%, 20%, 25%, 15%, 15%) 比例从 phase 1-6 采样。这分布给 phase 4-6(困难)更多权重,但保持 1-3(简单)的样本不被遗忘。

**比例的依据**:phase 4-6 的失败率高 → loss 更大 → 自然贡献更多梯度;但若样本数太少则梯度方差大。这个比例是经验起点,应在前 10 epochs 后看每 phase 的 loss 调整。

### 5.4 关于"用 RL 还是 BC"的决断

我**推荐 BC + DAgger,不推荐 RL**。理由:

1. **专家求解器可用**:RFC 给的三种 solver 是高质量监督源。BC + DAgger 数据效率远超纯 RL。
2. **小模型 RL 收敛慢**:PPO/Q-learning 在 ~300K 参数小模型上收敛要 ~10⁷-10⁸ 环境交互,而我们仿真器是 Python(慢);BC 只需 ~10⁵-10⁶ 样本,且都是离线生成。
3. **信用分配难**:推箱子是稀疏奖励长 horizon 任务,RL 信用分配代价高。BC 把这个交给求解器(它已经做了)。
4. **并不放弃自我对弈**:DAgger 本质上是"在自己策略下采样状态,用专家修正"——已经包含自我对弈的好处,但不需要解决信用分配。

**例外**:如果在 DAgger 后 phase 6 win rate 仍低于 85%,可考虑用 PPO **微调最后 1-2 个 epoch**,以求解器作 reference policy(KL 约束)。但这是 plan B,不是默认。

| 项 | 值 |
|---|---|
| Optimizer | AdamW |
| 学习率 | 3e-4(cosine 到 3e-5) |
| Weight decay | 1e-4 |
| Batch size | 256(GPU)/ 128(若显存紧) |
| Epoch(BC) | 80-120 |

## 6. 部署可行性 (Deployment Feasibility)

### 6.1 推理时延逐层估计

| 层 | input shape | output shape | INT8 MACs | 估时(M7@600MHz, 1.2 GMAC/s INT8 SIMD) |
|---|---|---|---|---|
| Stem 3×3, 38→48 | 38·10·14 | 48·10·14 | 38·48·9·140 = 2.30 M | 1.9 ms |
| Block 1 (×2 conv 48→48) | 48·10·14 | 48·10·14 | 2·48·48·9·140 = 5.81 M | 4.8 ms |
| Block 2 (同) | | | 5.81 M | 4.8 ms |
| Block 3 (d=2) | | | 5.81 M | 4.8 ms |
| Block 4 (d=2) | | | 5.81 M | 4.8 ms |
| Block 5 | | | 5.81 M | 4.8 ms |
| Block 6 | | | 5.81 M | 4.8 ms |
| Policy head 1×1 48→5 | 48·10·14 | 5·10·14 | 48·5·140 = 33.6 K | <0.1 ms |
| Value head | 48 | 1 | 48 | <0.1 ms |
| Aux heads(死锁 + 进度) | | | <100 K | <0.1 ms |
| **总计 MACs** | | | **~37.2 M** | **~31 ms** |

**推理时延估计:~31 ms**(48 ch 版本),**~52 ms**(64 ch 版本)。

64 通道版本**已经超出** 50 ms 预算,所以默认 **48 通道**。如果实际测试 48 ch 性能不够,首选**减少 block 数**(比如 5 个)而不是把通道加到 64,因为减 block 既减 MACs 也减参数。

**这些数字的不确定性**:
* CMSIS-NN 实测吞吐可能比 1.2 GMAC/s 高(2 GMAC/s)或低(800 MMAC/s),取决于 cache miss 率和数据布局。
* Dilation > 1 的卷积 cache 局部性差,实际可能慢 1.3-1.5×。
* SDRAM 带宽有限,对 weights 重复加载有 penalty。

**强烈建议**:在做架构最终决断前,**先在真机上跑一个空架(对应规模的随机权重 conv 网络)**测实际时延。这是 1-2 天的事,但能省掉后期返工。

### 6.2 内存预算

**Tensor arena**(中间 activation):

* Block 输入: 48·10·14 = 6720 elements × 1 byte = 6.6 KB
* Block 内部最大: 48·10·14·2(双 buffer)= 13.4 KB
* Stem 输入(38·10·14)= 5.3 KB
* 加上 padding / alignment ~30 KB

**Tensor arena 总估计:~40-80 KB,远低于 2 MB 预算**。

**Weights**:48 ch × 6 blocks 版本约 265 K params × 1 byte = 265 KB。**远低于 500 KB 预算**——这是有意保留的安全余量,留给:
* INT8 量化的 per-channel scales (~1 KB)
* 量化校准统计 (~1 KB)
* 备用通道扩展空间(若实测 48 ch 不够,可扩到 56 ch ≈ 360 K params,仍在预算内)

### 6.3 INT8 量化策略

TFLite Micro 的 INT8 量化采用 *per-tensor* 或 *per-channel* scale。我推荐:

* **权重 per-channel quantization**:每个输出 channel 单独 scale。这对 conv 几乎免费,精度损失最小。
* **Activation per-tensor quantization with PTQ + calibration**:用 100-500 张代表性图像(覆盖各 phase)做 activation range 校准。
* **不用 QAT(quantization-aware training)**,除非 PTQ 损失 >2 pp。QAT 训练成本翻倍,通常 PTQ + 校准就够。

**预期量化损失**:

* fp32 → INT8 的 win rate 下降在 0.5-2 pp 之间,RFC §6.4 的 ≤2 pp 预算可达。
* 主要敏感点:**softmax 之前的 logit range** 与 **value head 的输出**。这两处建议增加量化精度(用 INT16)或在 fp32 下做。
* 若动作合法掩码用 −∞,需要在 INT8 下用一个大负数(如 −127)模拟。

### 6.4 与 YOLO + 传统 CV 共存

RFC §4.3 提到 MicroPython + TFLite Micro 单进程串行。则:

```
Frame N:
  ├─ 50 ms  CV: AprilTag 解码 + 颜色分割
  ├─ 60 ms  YOLO: 推断箱子图案 / 数字(若进入 FOV)
  ├─ 35 ms  Decision: 本架构推断
  ├─ 25 ms  BFS 路径规划 + 低层动作下发
  └─ 30 ms  其他 (frame I/O, MCU 调度)
  ===========
   200 ms 总,5 Hz
```

这个预算**勉强可达 5 Hz**。建议:

1. **YOLO 与 Decision 不必每帧都跑**:YOLO 仅在新格出现 + FOV 内有未识别实体时跑;Decision 只在车到达"格中心"事件时跑(即每完成一格物理移动)。这能把平均频率从 5 Hz 降到 ~2 Hz,反而更稳。
2. **Decision 与 BFS 可以并行**:M4 副核(OpenART Plus)或 DMA 让 BFS 与 Decision 不互相阻塞。
3. **避免 1-step lookahead 在主循环**:它需要 K+1 次前向 = 150-180 ms,会破坏 5 Hz。把 lookahead 改成"仅在策略 top-1 置信度 < 阈值时启动",平均不影响吞吐。

### 6.5 测试与调试方案

部署前的离线测试:
1. 在 PC 上把训练好的 fp32 模型导出为 TFLite fp32,跑 1 万张图测 win rate(基线)
2. 量化为 INT8,在 PC 的 TFLite 解释器上跑同样 1 万张图,对比 win rate(量化损失)
3. 在 OpenART mini 实物上跑 100 张图,验证 (a) 时延 (b) 输出与 PC INT8 一致(确认 op 实现一致)
4. 只在 (3) 通过后,才上整车做 e2e 测试

这个 4 步流程把"模型对不对"和"硬件部署对不对"分开,失败时定位极快。

## 7. 风险点与备选方案 (Risks & Alternatives)

我对自己的方案不抱无条件信心。下面诚实列出**最可能失败的环节**及备选。

### 7.1 风险 #1:GroupNorm-fold 后量化精度下降过多

§3.3 提议的 GN + 后期 fold 是技巧性方案,不像普通 BN-fold 那样 well-tested。最坏情况 fp32 → INT8 精度下降 >3 pp,违背 RFC §6.4。

**备选 A**(优先):彻底放弃 normalization,只用 He init + 残差。训练可能会慢 30-50%,但部署完全干净。这在小图小模型上经验上是 OK 的。

**备选 B**:用 **Fixup Initialization** (Zhang et al. 2019)——专为无 BN 设计的初始化,残差路径上加固定的 scalar 缩放参数。这些 scalar 可以折到 conv weights 里,完全免费部署。

我推荐:**默认走备选 A 或 B,只有当训练发散/严重慢时才用 GN-fold**。

### 7.2 风险 #2:DAgger 在多箱场景上专家求解器太慢

Phase 6 的 BestFirst 在难图上 5-30 秒。DAgger 一轮要求解 ~1000 个状态 → 1.5-8 小时单核。这本身可接受,但若需要 5 轮 DAgger,总耗时 7-40 小时。

**备选**:
1. **多核并行**:训练机有 20 核 CPU,理论 20× 加速,实际 ~10×。
2. **分阶段 DAgger**:phase 1-3 用 IDA*,phase 4-5 用 BestFirst,phase 6 用 AutoPlayer(虽然次优)。
3. **保留失败样本**:每轮只对 BC 模型失败的 episodes 调用专家(典型 100-300 条/轮,而非全 1000)。

### 7.3 风险 #3:部分可观测性导致策略震荡

车进入新 FOV 后状态突变(突然多了已识别 ID)→ 策略可能突变 → 路径不连续。这会导致车反复横跳,虽然每步都"局部最优"。

**备选**:推理时维持一个**软状态平滑**——上一时刻的策略 logits 与当前的混合 0.3 + 0.7,提供连续性。或者引入"承诺机制":一旦选定推某箱子,在它被推入死角或推到目的地之前,下次决策只在该箱子的下一步候选里选(除非死锁概率超阈值)。这是个混合规则的 trick,经验上对 BC 策略有效。

### 7.4 风险 #4:48 通道不够表达 phase 6 的复杂度

如果 48 ch 在 phase 6 上 win rate 卡在 80%,而 64 ch 超出时延预算,怎么办?

**备选 A**:**减深度增宽度**:用 4 个 block × 64 ch ≈ 200 K params,时延 ~25 ms。如果感受野够(用更多 dilation)可能更好。
**备选 B**:**蒸馏**:训一个大教师网(无部署约束)在 GPU 上达到 95% phase 6,然后做 teacher-student 蒸馏到 48 ch 模型。教师的 soft target 比 BC hard target 信息更丰富,通常能让 student 提升 2-5 pp。
**备选 C**:**两阶段模型级联**:第一个微型 CNN 判断"这一步策略是否高置信度",若是则用它;若否则触发更深的搜索(调用 AutoPlayer 在 FLASH 里实现)。这是 anytime 思路的工程化。

### 7.5 风险 #5:对角推送规则学不会

RFC §2.3 的对角推送是**唯一非平移等变的特殊规则**。它在数据中可能极稀少 → BC 学不到 → 实战遇到死角图就 fail。

**备选**:把对角推送做成**纯规则,不让网络学**。在动作合法性掩码里检测"当前是炸弹 + 与墙对角相邻 + 对角通路空"的格局,在该格的 logit 上加一个固定 bias(比如 +5),让规则在该情境下被自动选中。零参数,绝对正确,绕过了 BC 的稀疏数据问题。

这又回到了 §4.5 的哲学:**逻辑可枚举的事不让网络学**。

### 7.6 风险 #6:CMSIS-NN 的 dilated conv 实现性能差

CMSIS-NN 对 dilated convolution 的 INT8 实现历史上不如 dilation=1 优化得好(数据局部性差、cache miss 多)。最坏情况实际时延比 §6.1 估计高 1.5×。

**备选**:把 §3.1 的 block 3-4 的 dilation=2 替换为 **stack of dilation=1 但深度+1**——感受野会减小一点(从 35 → 27),但仍 > 棋盘大小。或用 **5×5 conv 一层** 替代两个 3×3 dilation=2 conv,5×5 在 CMSIS-NN 上有 dedicated kernel,更快。

### 7.7 综合 fallback 路径

如果所有上面的备选都不够,我**最保守的 fallback 是**:

> **3 块 ResBlock × 32 通道,无 dilation,带所有距离场输入,无辅助头,纯 BC + DAgger 训练,无 lookahead。**
>
> 参数量:~70 K,INT8 ~70 KB
> 推理时延:~10-15 ms
> 预期 phase 6 win rate:75-85%(达不到目标但稳定可部署)

这是 win rate 的下界。如果它做不到 75%,说明本设计哲学(空间 CNN + BFS 输入)整体有问题,需重新思考——但我**评估这个概率 < 10%**,因为其他棋盘类任务(围棋、推箱子的 PIO benchmark)上类似架构都已被验证。

## 8. 与当前 Baseline 的差异点(总结表)

| 维度 | RFC §7.2 baseline | 本提案 | 改动收益(估计) |
|---|---|---|---|
| 状态表示 | 扁平向量 + 部分 2D | 全 2D 空间张量,38 通道 | +5-15 pp 多箱 win rate(因平移等变) |
| 动作输出 | 54 类 softmax (扁平) | H×W×5 全卷积空间策略 | +3-8 pp(因等变性 + 动作-状态一致性) |
| BFS 距离场使用 | 仅作 path planning 后处理 | **作为输入通道** | +5-10 pp(等效深度翻倍) |
| 死锁处理 | 无显式机制 | 辅助头 + 训练标签 | -10-20 pp 死锁失败率 |
| 训练范式 | BC | BC + DAgger + 值头自举 | +3-8 pp(distribution shift 修正) |
| 量化损失 | 不详 | PTQ per-channel calibrated, 估 ≤2 pp | 可控 |
| 推理 | branch search(浅层) | 1-step lookahead with value head(可选) | +1-3 pp,且更便宜 |
| 对称性利用 | 无 | D₄ 数据增广 | +1-3 pp |
| ID 排除推理 | 模型自学 | 显式预处理 | 预处理 → 让模型负担减小,间接 +1-2 pp |

**总叠加收益估计**:phase 4 从 40-50% → 80-90%,phase 5 从 60-75% → 90-95%,phase 6 从 50-80% → 88-93%。

注意这些数字是**乐观估计**:实际叠加常有抵消(不同改进解决相同失败模式)。**保守估计**:phase 4-6 全部达到 ≥85%,phase 1-3 接近 100%。

## 10. 结论

本设计的核心论点用一句话概括:

> **在 14×10 的小棋盘 + 500 KB INT8 + 50 ms 时延约束下,正确的架构是把 BFS 距离场作为输入通道、用全卷积小型 ResNet 输出空间策略图,辅以死锁/进度的密集监督头,训练范式用 BC + DAgger + 值头自举的三段式。**

这套设计的归纳偏置选择都有底层数学论证:平移等变(样本效率 ×140)、D₄ 增广(8×)、ID 重命名对称(避免学错符号绑定)、距离场作为信息瓶颈的近似充分统计量(深度 -50%)。每一个选择都对应一个具体的失败模式或资源约束。

它不是也不会是 SOTA 在算力无限场景下的"最先进"方案——本来就不是这个目标。它是**在 RFC §4 列出的硬约束下、对 §6.2 目标 win rate 最有把握的方案**。

如果让我重新设计一次,我能想到的唯一会动摇当前方案的因素是:**如果未来 OpenART 系列加上了 NPU 支持基本 attention**——那时一个微型 GNN over (box, target) 节点的方案会更优雅。但当前规格上,本方案是我能给出的最佳答复。

## 附录 B:参数量精确推导

**48 通道版本(推荐)**:

* Stem: $38 \times 48 \times 9 = 16{,}416$
* 6 blocks × $(48 \times 48 \times 9 + 48 \times 48 \times 9) = 6 \times 41{,}472 = 248{,}832$
* Policy head: $48 \times 5 = 240$
* Value head: $48 + 1 = 49$
* Deadlock head: $48 \times 1 = 48$
* Progress head: $48 \times 1 = 48$
* **总计:265,633 params ≈ 266 K**
* **INT8: 266 KB,加 per-channel scales(≈1 KB),约 267 KB**

**FLASH 占用估计**:267 KB(weights) + ~50 KB(模型 metadata + tflite header) ≈ 320 KB,远低于 500 KB 预算。

**RAM 占用估计**:
* Tensor arena: ~80 KB
* Stack + heap: ~100 KB(MicroPython 开销)
* **总计:~180 KB,低于 2 MB 预算**
