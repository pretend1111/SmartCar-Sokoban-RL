方案：SAGE-Push — 面向 MCU 的“符号约束 + 神经排序 + 浅层搜索”推箱子决策架构

我建议不要把这个赛题做成“端到端从状态直接 softmax 到 54 类动作”的问题，而是做成：

经典算法负责可证明的离散约束、路径、可达性、死锁、FOV 信念更新；神经网络只学习最难手写的部分：在当前局面下，哪个合法高层候选动作最值得执行。

这样可以把模型从“学习 Sokoban 全部物理规则”中解放出来，让它学习专家搜索器中真正昂贵的全局优先级判断。赛题地图只有 14×10 有效区域、箱子最多 5 个、炸弹最多 3 个，但硬件只有 Cortex-M MCU、INT8、≤500 KB 权重、≤50 ms 单次决策推理，因此这种候选动作排序器比通用 CNN+MLP 更符合问题结构和部署约束。RFC 中给出的核心约束是：OpenART mini/Plus 无 GPU/NPU、决策模型 INT8 权重 ≤500 KB、tensor arena ≤2 MB、OpenART mini 单次推理 ≤50 ms，且只允许 Conv2D / DepthwiseConv2D / FullyConnected / ReLU / Reshape / Softmax / Concat / Add / Pooling 等 TFLite Micro 友好算子。

RFC_neural_arch_design

 TensorFlow Lite for Microcontrollers 官方文档也强调它面向微控制器这类低内存设备，只支持有限算子子集，并推荐小模型与量化部署；CMSIS-NN 则是 Arm Cortex-M 上用于降低内存占用、提升 NN kernel 效率的官方库。

0. 总体判断

推荐架构名称：SAGE-Push
全称：Symbolic-Aided Graph/Set-Equivariant Push Policy

它由四个部分组成：

符号状态层：维护地图、实体、已知/未知 ID、FOV 记忆、ID 排除信念矩阵、BFS 距离场、死角与推送距离场。

候选动作生成器：枚举所有合法高层动作，包括推箱、推炸弹、爆破、探索未知 ID、入库。

神经候选排序器 PushRank-μ：一个约 140k 参数的 INT8 小模型，对最多 64 个候选动作进行等变打分，并输出局面价值。

浅层神经引导搜索：beam width 4、depth 2–4 的小搜索，用网络评分 + 符号模拟选择第一步动作。

部署时，小车不需要每 200 ms 都运行完整决策网。决策网只在“到达格点中心、完成一次推送、识别状态变化、炸弹爆炸、箱子消除、探索动作完成”后运行；中间的物理运动由 BFS 路径与电机闭环执行。这与 RFC 中已有的“高层决策 + BFS 低层路径”工程拆分兼容，但我建议把神经输出从固定类别 softmax 改成候选动作集合上的排序 / 价值估计。

3.3 为什么不是让网络直接学 BFS / 推链 / 爆炸？

因为这些规则是确定、低成本、可验证的。让神经网络学习它们只会浪费参数和样本，并引入非法动作风险。Sokoban 类问题本身有强组合复杂性，经典研究已证明许多推箱/运动规划变体具有很高计算复杂性；因此应该把神经网络用于启发式评估，而不是替代所有约束推理。

6. 部署方案
6.1 MCU 推理路径

部署循环：

        ↓
YOLO 仅在需要识别 ID 时运行
        ↓
符号状态更新：FOV、ID、belief、墙体、实体
        ↓
BFS / deadlock / reverse push distance
        ↓
生成 ≤64 个候选动作
        ↓
PushRank-μ INT8 前向
        ↓
浅层 beam search
        ↓
输出 macro 动作
        ↓
BFS 生成低层路径
        ↓
电机闭环执行


关键调度建议：YOLO 不应每帧无条件运行。当所有剩余 ID 已知，或当前动作不需要识别时，跳过 YOLO，把时间让给运动控制与决策。RFC 中说明 CV、YOLO、决策模型在主循环串行排队，目标周期 ≥5 Hz，因此这种按需调度非常重要。

6.3 推理时延估计

以 5.7M MAC、约 140k INT8 权重估算：

平台	预期 PushRank-μ 单次前向
OpenART mini / MIMXRT1064 600 MHz	8–25 ms，保守上界 <50 ms
OpenART Plus / MIMXRT1176 M7 1 GHz	5–15 ms

这不是最终承诺值，必须用实物 profiling 校准；但从模型规模看，留给候选生成、BFS 和浅层搜索的余量是足够的。CMSIS-NN 官方定位就是为 Cortex-M 处理器最大化神经网络 kernel 性能并降低内存占用。

6.4 与 YOLO 共存

考虑到 RFC 提到历史上同时加载多个 TFLite 模型可能不稳定，我建议两级部署策略。

RFC_neural_arch_design

首选：YOLO 用 TFLite Micro，PushRank-μ 用手写 CMSIS-NN C 推理

PushRank-μ 的结构非常简单，本质是 Conv/FC/ReLU/Pool。可以训练后导出 int8 权重，写成固定 C 数组，用 CMSIS-NN kernels 直接推理。这样避免第二个 TFLite interpreter / arena。模型仍保持 TFLite Micro 可表达性，用于仿真和验证。

备选：单 interpreter 分时加载

如果必须 TFLite Micro 运行 PushRank-μ，则：

决策模型 arena 单独固定；

YOLO 与决策不在同一帧运行；

已知所有 ID 后卸载或暂停 YOLO；

决策模型输入 buffer 静态分配；

所有临时 BFS 数组复用固定内存池。

8. 风险点与备选方案
风险 1：phase 6 炸弹策略仍然不稳定

原因：炸弹改变地图拓扑，爆破动作短期可能看起来绕远，但长期必要。

备选：

给炸弹单独训练一个 BombValue 辅助头；

对每个爆破候选计算连通分量增益；

离线生成“爆破必要性”标签；

部署搜索中对爆破候选加一层 lookahead；

如果模型低置信度，调用手写规则：优先爆破能连接箱子-目标 push-distance 不可达区域的墙。

风险 2：探索策略过保守或过激进

这不是坏事。MCU 任务的目标不是展示端到端神经优雅性，而是在硬约束下赢比赛。真正危险的是规则特征覆盖不到某些长程组合。

增加 depth-4 beam search；

增加 value head 权重；

为 phase 4–6 专门 oversample 失败局面。

风险 4：TFLite Micro 与 YOLO arena 冲突

备选：

PushRank-μ 改为 CMSIS-NN 手写推理；

YOLO 只在探索动作中运行；

决策模型和 YOLO 分时加载；

把 PushRank-μ 宽度从 96 降到 64，参数降到约 72k；

移除 grid encoder，仅保留候选排序器，权重可降到 50k 以下。

风险 5：量化后 top-1 抖动

备选：

QAT；

score margin loss；

int16 accumulator；

部署用 top-4 beam 而不是单 argmax；

若两个候选分数差小于阈值，选择符号启发式更安全的那个：不死锁、信息增益高、路径短、可撤销。

	​


这是比通用 hybrid CNN+MLP 更适合该赛题的原因：

问题结构上，推箱子是强离散约束 + 小对象集合 + 长程规划，不是普通图像分类。

硬件上，MCU 不适合大 CNN、Transformer、RNN 或端到端世界模型。

学习上，候选排序比固定 action softmax 更抗实体排列、箱子数量变化和 ID 配对组合爆炸。

安全上，非法动作、死锁、FOV、推链、爆炸由符号层保证，不交给网络猜。

泛化上，DeepSets/关系归纳偏置、规划特征、DAgger、hard negative 都直接针对 baseline 在 phase 4–6 的失效模式。

部署上，约 140k INT8 参数、约 5.7M MAC、单模型 TFLite/CMSIS-NN 友好，满足 ≤500 KB、≤2 MB、≤50 ms 的约束空间。

最关键的一句话：不要训练一个“小脑袋”去凭感觉输出动作类别；要训练一个“神经启发式函数”去给符号生成的合法候选动作排序。
